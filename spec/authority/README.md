# Delegated Spending Authority

This bundle defines identity, the `SpendingAuthorizationCredential`, securing
mechanisms, and the issuer-trust framework for autonomous-agent payments. The
[AVP-Micro Payments](../payments/) specification depends on this one.

## Canonical URLs (registration pending)

The context and vocabulary are authored for these `w3id.org` URLs; registering
the redirects is a prerequisite for cross-implementation interoperability:

- Context: `https://w3id.org/spending-authority/v1` → `context/v1.jsonld`
- Vocabulary namespace: `https://w3id.org/spending-authority/v1#`

DSA objects (credentials, trust configuration) use the **3-entry** context array:

```json
["https://www.w3.org/ns/credentials/v2",
 "https://w3id.org/security/data-integrity/v2",
 "https://w3id.org/spending-authority/v1"]
```

## Test vectors

`test-vectors/` contains five files:

| File | Description |
|------|-------------|
| `spending-authorization-credential.json` | Signed `SpendingAuthorizationCredential` (issuer → agent) |
| `merchant-credential.json` | Signed `MerchantCredential` (issuer → payee) |
| `payment-capability-credential.json` | Signed `PaymentCapabilityCredential` (wallet → agent) |
| `trusted-issuers.json` | Example wallet trust-configuration (not a signed wire message) |
| `dids.json` | Test DID values for all four principals (test keys only) |

The `spending-authorization-credential.json` vector is **embedded** by the
AVP-Micro Payments chain: `02-payment-authorization.json` and
`06-session-budget-authorization.json` include it inside a Verifiable
Presentation to demonstrate that the payer holds a valid authorization from a
trusted issuer.

## Service-category vocabulary

`vocab/agent-service-categories.ttl` is a SKOS concept scheme describing
agent-service categories. Its scheme IRI is
`https://w3id.org/avp-micro/cat/scheme/AgentServiceCategory`; each concept uses
the `https://w3id.org/avp-micro/cat#` namespace (prefix `cat:`). Generated by
`vocab/generate_agent_service_skos.py`.

## Securing mechanism

Mandatory-to-implement: W3C Data Integrity `DataIntegrityProof` with cryptosuite
`eddsa-jcs-2022`, over `did:key` (Ed25519 / `Multikey`). See the spec
"Securing mechanisms" section.
