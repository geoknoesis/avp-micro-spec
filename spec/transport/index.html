This specification defines the normative HTTP/REST binding for AVP-Micro: how an agent (payer) and a service (payee) discover each other and run the offer → quote → authorize → execute → settle → receipt flow over HTTP, using an HTTP 402 "Payment Required" challenge. It transports the existing AVP-Micro payment objects; it defines no new economic semantics. It is the prerequisite that lets two independent implementations interoperate.

Overview

AVP-Micro specifies messages (DSA credentials, payments, disputes, settlement). This bundle is the wire layer. It adds four object types and an error-code vocabulary, all in the namespace https://w3id.org/avp-micro/transport/v1# (prefix txp:), and an OpenAPI 3.1 contract for the HTTP surface.

Discovery

GET /.well-known/avp-micro returns a payee-signed ServiceDescription advertising endpoints, acceptedCredentialIssuers, acceptedSettlementRails, and supportedBundles. Clients SHOULD fetch it before transacting.

The HTTP 402 challenge flow

  1. Agent requests a gated resource.
  2. Payee replies 402 Payment Required with WWW-Authenticate: AVP-Micro and a body { challenge: PaymentChallenge, quote: PaymentQuote }. The PaymentChallenge carries quoteDigest (= jcs_digest(quote)), a server challenge nonce, and expires.
  3. Agent verifies the quote, builds a PaymentAuthorization (which embeds the SAC verifiable presentation), and wraps it in an AuthorizationSubmission that echoes the challenge, references the authorization by IRI + authorizationDigest, and carries an idempotencyKey.
  4. Agent retries with Authorization: AVP-Micro <submission> and Idempotency-Key.
  5. Payee verifies the quote binding, SAC policy, challenge freshness, and idempotency, settles, and returns 200 with the resource and a PaymentReceipt — or a 4xx/402 with a ProblemDetails.

The echoed challenge binds the submission to the verifier and request that issued the 402. A captured authorization cannot be replayed to a different verifier — closing the presentation-replay gap identified in the security review.

Explicit-quote, streaming, and async settlement

Non-gated clients MAY use POST /quote, POST /authorize, GET /receipt/{id} directly. Streaming reuses the payments session objects via POST /session, /session/{id}/budget, /session/{id}/accruals, /session/{id}/extend, /session/{id}/close. When settlement is not instant, execute returns a pending PaymentExecution and a Location: /settlement/{id}; polling that endpoint returns the execution and then the SettlementProof as finality advances. A callbackUrl MAY be supplied for a webhook instead of polling.

Idempotency

Idempotency-Key (mirrored in AuthorizationSubmission.idempotencyKey) makes retries safe. The payee MUST return the same execution/receipt for a repeated key, and MUST return 409 idempotency-conflict if the key is reused with a different body.

Error model

Every non-success response is a ProblemDetails whose type is an error-code IRI in txp:ErrorScheme (see vocab/errors.ttl).

Security considerations

  • Challenge freshness. The payee MUST reject a submission whose echoed challenge is unknown, expired, or already consumed. This binds the SAC presentation to a single verifier and request.
  • Quote binding. The payee MUST verify quoteDigest == jcs_digest(quote) and that the quote's requestHash matches the attempted request.
  • Transport security. All exchanges MUST use TLS; AVP-Micro proofs authenticate payloads but do not replace channel encryption.

Conformance & artifacts

The JSON-LD context, RDFS/OWL vocabulary, SKOS error scheme, JSON Schema, SHACL shapes, OpenAPI contract, and signed test vectors are normative companions in this bundle directory. The shared harness (spec/generate.py, verify.py, validate.py) generates and validates them.
