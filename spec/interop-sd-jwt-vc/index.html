This interop profile defines a normative, bidirectional bridge between an AVP-Micro verifiable credential expressing delegated spending authority — the Spending Authorization Credential of [[DSA]], optionally carried in a [[AVP-MICRO-PAYMENTS]] PaymentAuthorization — and an SD-JWT-VC mandate of the kind used by Mastercard/Google Verifiable Intent [[?VERIFIABLE-INTENT]] and Google AP2 [[?AP2]]. The two ecosystems share the concept of a cryptographically verifiable, delegated authority chain but agree on no wire format, proof suite, or identifier scheme: AVP-Micro secures JSON-LD credentials with W3C Data Integrity (eddsa-jcs-2022) over DIDs, whereas SD-JWT-VC secures JOSE tokens with ES256 over iss/sub identifiers. This profile specifies how spending authority crosses that line without a privileged intermediary becoming the root of trust: a claim mapping, an identity binding, two embedding envelopes, and the verification rules a relying party applies to each.

This document is an unofficial editor's draft prepared as a candidate contribution to a W3C Community Group. It is an interop profile / binding, not part of the AVP-Micro core protocol: it depends on both [[DSA]] and [[AVP-MICRO-PAYMENTS]] and on the external [[SD-JWT-VC]] family, and it is versioned and published as its own bundle so that an external revision never destabilizes the core specifications. It is published with W3C ReSpec and does not represent the consensus of the W3C Membership or of any Working or Community Group. It MUST NOT be cited as anything other than work in progress.

Introduction

Several agent-payment ecosystems are emerging concurrently. AVP-Micro expresses delegated spending authority as a [[DSA]] Spending Authorization Credential — a W3C [[vc-data-model-2.0]] credential in JSON-LD, secured with a Data Integrity proof and identified by [[did-core]] DIDs. Mastercard/Google Verifiable Intent [[?VERIFIABLE-INTENT]] and Google AP2 [[?AP2]] express the equivalent authority as an SD-JWT-VC mandate: a layered SD-JWT / KB-SD-JWT chain [[SD-JWT]] [[SD-JWT-VC]] secured with JOSE ES256 [[RFC7515]] [[RFC7518]] and identified by iss/sub URIs and an opaque key identifier.

The two describe the same shape — an issuer-rooted chain delegating bounded spend to an agent, verifiable by any party in the chain without contacting the issuer — but they share no encoding, proof suite, or identifier scheme. This is the long-standing JSON-LD/Data-Integrity versus JOSE/SD-JWT divide, surfacing in agentic commerce. Because the well-resourced corner of the space has chosen SD-JWT-VC, AVP-Micro authority that cannot travel into that ecosystem risks being islanded; and conversely, AVP-Micro relying parties benefit from being able to accept mandates that originate there. This profile makes both directions possible.

The central design constraint is cryptographic and unavoidable: a Data Integrity proof signs the JCS-canonical [[RFC8785]] JSON-LD bytes, while an SD-JWT signature signs the JOSE serialization. Translating the claims from one envelope to the other changes the signed bytes, so the original signature can never verify over the translated form. Every rule in this profile follows from that fact.

How it works

Authority crosses the boundary in one of three bridge modes, distinguished by who, if anyone, becomes a new trust root:

  • Proof-preserving (default). The original credential and its original proof are carried, intact, inside the target envelope; the relying party verifies both the envelope (for transport, typing, and holder binding) and the embedded original proof (for authority). The bridge is a pure transcoder and adds no trust.
  • Co-issued. Where issuance is under the issuer's control, the issuer natively signs both representations over the same claims at creation. No intermediary exists. This is the strongest mode and is preferred wherever applicable.
  • Attested. A named bridge verifies the source and re-issues a new credential in the target format signed by its own key. The bridge becomes a trust root; relying parties MUST independently and explicitly trust it. This mode is a fallback for relying parties that cannot perform cross-stack verification.

The remainder of this document specifies the identity binding (), the claim mapping (), the chain/layer mapping (), status (), the two envelopes for export and import (, ), the relying-party verification algorithm (), and what does not round-trip ().

Identity binding

The two stacks identify parties differently and asymmetrically. AVP-Micro uses DIDs ([[DID-KEY]] mandatory-to-implement, [[DID-WEB]] permitted), with the agent holder key expressed implicitly: the agent's key is credentialSubject.id, and the same key signs the PaymentAuthorization — i.e. AVP-Micro already performs SD-JWT-style holder key binding, using DIDs and Data Integrity rather than cnf [[RFC7800]] and a key-binding JWT. SD-JWT-VC uses an iss issuer URI, a sub subject identifier, an opaque kid, and a cnf holder key.

Export (AVP-Micro → SD-JWT-VC)

This direction is near-lossless, because a DID is a URI. An exporter MUST:

  • set iss to the credential issuer DID;
  • set sub to credentialSubject.id (the agent DID), and set cnf.jwk to the agent's public key expressed as a JWK (an OKP/Ed25519 JWK derived from the agent did:key multibase, or the verification method published in the agent's DID document);
  • set kid to the agent verification-method DID URL.

A consumer that understands DIDs MAY resolve these values as DIDs; a consumer that does not treats them as opaque URIs. No authority information is lost.

Import (SD-JWT-VC → AVP-Micro)

This direction is lossy in general: iss/sub are opaque with no mandated key-resolution method, and SD-JWT-VC keys are ES256 (P-256), which the AVP-Micro mandatory cryptosuite eddsa-jcs-2022 (Ed25519 only) cannot express. To enable proof-preserving import () this profile defines the following did:web binding convention, which a conforming importer operating in proof-preserving or co-issued mode MUST rely upon:

  • The source iss (and, where the subject is itself an issuer, sub) MUST be, or deterministically map to, a [[DID-WEB]] identifier whose DID document publishes the signing key as a verification method of type JsonWebKey bearing the ES256/P-256 public key.
  • The source kid MUST resolve to that verification method (by fragment or by JWK thumbprint).
  • The importer verifies the embedded JOSE proof against the resolved P-256 verification method (); it does not re-sign.

Where the did:web binding convention cannot be satisfied (no resolvable DID, or a key type that cannot be published), proof-preserving import is not possible and the importer MUST either refuse or fall back to attested mode (), never silently emit an object that claims authority it cannot substantiate.

Chain and layer mapping

The layers of an SD-JWT-VC mandate correspond to AVP-Micro objects as follows. The correspondence is exact for the autonomous-agent case (the agent-payments case this profile targets):

SD-JWT-VC layerSignerAVP-Micro analogue
L1 — identity + holder-key binding Issuer [[DSA]] Spending Authorization Credential issuance (issuer → credentialSubject + key)
L2 — purchase intent User / principal Standing: folded into the mandate (the delegation is the pre-authorized intent). Interactive: no direct analogue — see below.
L3 — agent action on a specific purchase Agent (holder key) [[AVP-MICRO-PAYMENTS]] PaymentAuthorization proof: the agent DID (= credentialSubject.id) signs over quoteDigest, amount, serviceRequestHash, nonce, expires

The L1↔L3 key binding of SD-JWT-VC (the key-binding JWT chained to the issuer layer) is mirrored in AVP-Micro by the PaymentAuthorization embedding the credential and being signed by the same key named as credentialSubject.id. The key-binding semantics line up; only the encoding differs.

SD-JWT-VC's interactive L2 — a fresh per-purchase human approval — has no AVP-Micro analogue, because AVP-Micro's trust model is standing delegation: the human's intent is expressed once, ahead of time, by issuing the mandate. An importer MUST mark a chain that carries an interactive L2 as imported with an advisory: the resulting EmbeddedSdJwtVcMandate represents the delegated envelope, not a per-purchase human action, and a relying party that specifically requires fresh human intent cannot obtain that guarantee from the imported form. For autonomous-mode mandates (where L2, if present, is itself a standing mandate), this gap does not arise.

Status and revocation

AVP-Micro uses BitstringStatusListEntry [[BITSTRING-STATUS-LIST]]; SD-JWT-VC uses the IETF Token Status List [[TOKEN-STATUS-LIST]]. Both index a bit into a hosted, compressed bitstring and map structurally. A conforming exporter or importer:

  • MUST translate the status reference but MUST NOT re-host the status list: the produced object points at the original issuer's status endpoint, so revocation remains controlled by the principal, not by the bridge;
  • when both references are present (as in co-issued mode), MUST keep them consistent; a verifier checking either is sufficient, and the two MUST agree.

A cross-stack verifier MUST check whichever status mechanism its stack understands and MUST treat a revoked status as fatal.

Export: AVP-Micro → SD-JWT-VC

An SD-JWT-VC carrying the AVP-Micro authority for use by an SD-JWT-VC / AP2 verifier.

In proof-preserving mode (default), an exporter produces an SD-JWT-VC that carries the original credential as a protected, non-disclosable claim: 

{
  "vct": "mandate.spending-authority.avp+embedded",
  "iss": "did:web:principal.example",
  "sub": "did:key:z6Mko688Vt1uZFQZh8Wvb8BnmenEqTajZwE2BugW85nWA65R",
  "cnf": { "jwk": { "kty": "OKP", "crv": "Ed25519", "x": "…agent public key…" } },
  "currency": "USD",
  "limits": { "per_txn": "0.05", "per_day": "5.00" },
  "allowed_payees": ["did:key:z6MkenPA2xcD5CGJVmkgejbr4LzBAa1e3GVCQiHFrXDgh5Vi"],
  "nbf": 1774483200,
  "exp": 1782345600,
  "jti": "urn:dsa:vc:spendauth:456",
  "status": { "status_list": { "idx": 94567, "uri": "https://issuer.example/status/3" } },
  "avp_vc": ""
}

The outer SD-JWT issuer signature provides JOSE-native typing, selective disclosure, and (via cnf) holder binding; the avp_vc claim carries the authority. The avp_vc claim MUST NOT be marked selectively-disclosable: the authority must always be present. An exporter producing the mandate.spending-authority.avp+embedded profile MUST set avp_vc to the byte-exact JCS-canonical form that the original Data Integrity proof covers.

In attested mode an exporter MAY omit avp_vc and instead rely on its own issuer signature over the mapped claims; the produced vct is then mandate.spending-authority.avp and the bridge is the issuer of record. In co-issued mode the original issuer signs this SD-JWT-VC natively over the same claims as part of issuance.

The mode determines the trust of the outer object:

  • proof-preserving: there is no Ed25519 proof to be had (the principal signed only JOSE), so the outer object carries no proof and is an unsigned projection. Its authority is the embeddedSdJwtVc, verified via the did:web binding convention (). An importer MUST set bridgeMode to proof-preserving and MUST NOT add a proof.
  • co-issued: the outer proof is a genuine eddsa-jcs-2022 proof by the principal's Ed25519 key over the mapped claims, produced at issuance — no bridge in the path.
  • attested: a named bridge re-signs the outer object; the bridge DID MUST appear in iop:attestingBridge and in proof.verificationMethod, and relying parties MUST independently trust it.

Cross-stack verification

A cross-stack verifier presented with a bridged object MUST apply all of the following. These rules are the security core of the profile.

  1. No-downgrade. For a proof-preserving object the verifier MUST verify the embedded original proof for authority and MUST NOT treat the outer envelope signature (the SD-JWT issuer signature on export, or the absence of an outer proof on import) as a substitute. The outer signature attests only transport, typing, and holder binding.
  2. Algorithm pinning. Ed25519 (Data Integrity) and ES256 (JOSE) MUST each be verified against the layer that legitimately uses it. A verifier MUST NOT accept one algorithm where the other is expected (algorithm confusion).
  3. Explicit bridge trust. A verifier MUST NOT honor an attested object unless the iop:attestingBridge DID is on an explicit, operator-configured trust list. Authority obtained in proof-preserving or co-issued mode requires no such list.
  4. Window intersection. When both stacks carry validity or replay bounds (validFrom/validUntil vs nbf/exp; nonce/jti; expires), the verifier MUST enforce the intersection (most restrictive). A bridged object MUST NOT be honored beyond the narrowest bound present.
  5. Status. The verifier MUST check status per and treat revocation as fatal.
  6. Holder binding. The verifier MUST confirm that the key exercising the mandate (the SD-JWT key-binding JWT signer, or the AVP-Micro PaymentAuthorization signer) is the holder key bound in L1 / named as credentialSubject.id.

Round-trip and losslessness

A conforming implementation MUST preserve every field in across a full round-trip in both directions (A→V→A and V→A→V) and MUST preserve verifiability of the embedded original proof. The following are inherently lossy and an implementation MUST surface (not silently drop) each:

  • Interactive L2. SD-JWT-VC fresh per-purchase human intent has no AVP-Micro equivalent (); flag on import.
  • Native selective disclosure. SD-JWT selective disclosure has no Data-Integrity equivalent. On export, selective disclosure MAY be introduced only in attested mode (the bridge re-issues); under proof-preserving mode the embedded credential is whole, so selective disclosure of the embedded authority is NOT available and MUST NOT be implied. On import of a partially disclosed SD-JWT, the resulting mandate is a subset view and MUST be flagged as partial.
  • P-256 keys. Not expressible in eddsa-jcs-2022; proof-preserving import depends on the did:web binding convention (), failing which the object cannot be imported proof-preservingly.

Security considerations

The dominant risk is a bridge that silently becomes a trust root. proof-preserving and co-issued modes add no trust by construction; attested mode does, and is constrained by rule 3 of (explicit trust list) and by mandatory iop:attestingBridge disclosure.

Downgrade and confusion. The no-downgrade and algorithm-pinning rules () prevent an attacker from substituting a cheaply-forged outer signature for the embedded authority, or from exploiting the two distinct signature algorithms. A verifier that checks only the outer envelope is non-conforming and unsafe.

Authority widening. Translation MUST NOT widen authority. The window-intersection rule and the decimal-string discipline of prevent a translation from extending validity or altering caps. Status references are re-pointed but never re-hosted (), so the principal retains revocation control across the bridge.

Replay. The per-purchase layer (SD-JWT key-binding JWT / [[AVP-MICRO-PAYMENTS]] PaymentAuthorization) carries the nonce, request binding, and short expiry that bound a mandate to a single action; the bridge transports these unchanged and the verifier enforces them.

Privacy considerations

The proof-preserving envelope embeds the original whole credential, which is not selectively disclosable (); a holder that needs to minimize disclosure into the SD-JWT-VC ecosystem must use attested re-issuance, accepting the bridge as a trust root in exchange for selective disclosure. Conversely, importing a partially-disclosed SD-JWT yields only the disclosed claims, which the importer MUST flag as a partial view rather than presenting it as a complete mandate. Identifiers (DIDs, iss/sub) are carried verbatim; deployments that treat agent or principal identifiers as correlatable SHOULD apply the same pairwise-identifier practices recommended by [[DSA]].

Internationalization considerations

This profile carries currency codes and decimal amounts as opaque strings and does not localize them; presentation and rounding are out of scope, exactly as in [[AVP-MICRO-PAYMENTS]]. Datetime conversions between RFC 3339 strings and NumericDate integers MUST preserve the absolute instant and use UTC.

Accessibility considerations

This profile defines data interchange between software agents and has no direct user interface. Tooling that surfaces bridge provenance (the bridgeMode and any attestingBridge) to operators should follow general accessibility guidance for that tooling.
