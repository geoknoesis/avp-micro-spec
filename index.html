AVP-Micro Specification Suite

This document provides an overview of the AVP-Micro specification suite: a set of open, standards-based specifications for delegated, verifiable spending authority for autonomous AI agents. The suite defines how an issuer grants bounded spending rights to an agent, how a merchant independently verifies those rights at transaction time, and how all objects in the payment lifecycle are cryptographically secured using W3C Verifiable Credentials 2.0, Decentralized Identifiers, and Data Integrity proofs.

A third document specifies a normative interoperability bridge to SD-JWT-VC credentials as used by Mastercard/Google Verifiable Intent and Google AP2.

This is an unofficial draft produced by Geoknoesis LLC. It has not been reviewed by a W3C Working Group. Feedback and contributions are welcome via the GitHub repository.

Introduction

Autonomous AI agents that spend money on behalf of users or organizations require a trust layer that answers three questions with cryptographic proof:

  1. Who authorized this agent to spend? — proven by a signed SpendingAuthorizationCredential.
  2. What are the spending terms? — limits, allowed payees, currency, and expiry encoded in the credential subject and enforced by verifiers.
  3. Is this specific purchase authorized? — confirmed by a PaymentAuthorization signed by the agent and bound to a merchant-issued quote.

AVP-Micro provides the vocabulary, credential formats, cryptographic securing mechanisms, and conformance test harness for this trust layer. It deliberately does not specify payment rails or settlement — those are pluggable substrates. The same credential grammar spans Lightning, Interledger, card, and stablecoin rails alike.

Design goals

Vendor-neutral trust anchor
Authority is proven by verifying an issuer's credential against a DID — not by trusting any specific network's ledger, token format, or settlement guarantee.
Open standards only
All securing mechanisms are built on ratified W3C specifications: [[VC-DATA-MODEL-2.0]], [[DID-CORE]], [[VC-DATA-INTEGRITY]], [[VC-DI-ECDSA]], [[BITSTRING-STATUS-LIST]], and [[JSON-LD11]].
Interoperable with SD-JWT-VC networks
The interop profile ([[AVP-INTEROP]]) provides a normative, bidirectional mapping to SD-JWT-VC credentials without collapsing either trust stack.
Conformance-testable
Every normative requirement is exercised by signed test vectors generated by a shared Python harness (generate.py) and verified by verify.py and validate.py.

Specification suite

The suite comprises four peer documents. Each ships a normative ReSpec specification, a JSON-LD 1.1 context, a JSON Schema, SHACL shapes, an OWL/RDFS vocabulary, and signed conformance test vectors. A reference protocol simulator exercises the full message flow end to end against a play-money ledger.

Delegated Spending Authority

The Delegated Spending Authority specification [[DSA]] is the identity and trust foundation. It defines:

  • The SpendingAuthorizationCredential — a W3C VC 2.0 credential whose subject is an agent DID, carrying per-transaction and daily amount limits, an allowed-payees list, currency, and expiry.
  • The MerchantCredential and PaymentCapabilityCredential for payee and wallet endorsement.
  • The mandatory-to-implement securing mechanism: DataIntegrityProof with cryptosuite ecdsa-jcs-2022 and DID method did:key (P-256 / Multikey, multicodec p256-pub).
  • Revocation via [[BITSTRING-STATUS-LIST]].
  • An Agent Service Category SKOS vocabulary for scoping spending authority by domain.

Namespace: https://w3id.org/spending-authority/v1#

AVP-Micro Payments

The AVP-Micro Payments specification [[AVP-PAYMENTS]] defines the full payment lifecycle on top of DSA credentials:

  • PaymentOffer — merchant-issued, signed offer with a normative pricing-model vocabulary (flat, per-call, tiered, composite).
  • PaymentQuote — merchant-computed, signed binding of price to a specific service request.
  • PaymentAuthorization — agent-signed approval of a quote, committing to economic terms and binding the agent's spending credential.
  • PaymentExecution and PaymentReceipt — wallet and merchant records closing the transaction.
  • UsageSession, UsageAccrual, and SessionBudgetAuthorization — streaming/metered payment objects for long-running or usage-billed services.

Namespace: https://w3id.org/avp-micro/v1#

SD-JWT-VC Interop Profile

The SD-JWT-VC Interop Profile [[AVP-INTEROP]] is a bridge/binding, not a third protocol pillar. A W3C Data Integrity proof signs JCS-canonical JSON-LD bytes; an SD-JWT signature signs JOSE serialization. Translating credentials from one envelope to the other changes the signed bytes — the original signature can never verify over the translated form. Authority therefore crosses the boundary in one of three bridge modes:

ModeMechanismTrust added by bridge
proof-preserving (default) Embed the original credential + proof; the verifier checks both None
co-issued Issuer natively signs both forms at creation time None
attested A named bridge re-signs in the target format Bridge becomes a trust root; must be explicitly trusted by relying party

Namespace: https://w3id.org/avp-micro/interop/sd-jwt-vc/v1#

Refunds, Reversals & Disputes

The Refunds, Reversals & Disputes specification defines the reverse value-flow on top of Payments — the voluntary path and the adversarial path, both converging on a wallet-signed reversal:

  • Refund — a payee-signed return of value (full or partial) against a receipt/execution.
  • Reversal and ReversalAcknowledgement — the wallet-signed settlement that undoes a charge (caused by a refund or an upheld dispute), and the payer's acknowledgement of it.
  • Dispute, DisputeEvidence, and DisputeResolution — the adversarial lifecycle: a payer raises a dispute, both sides submit evidence, and the payee or a named arbiter resolves it (upheld / partial / rejected / withdrawn).

Namespace: https://w3id.org/avp-micro/disputes/v1#

Conformance harness

A shared Python harness at the repository root generates all signed test vectors and validates every bundle. Dependencies are installable via pip install -r requirements.txt.

Commands

python spec/generate.py
python spec/verify.py    # must report PASS
python spec/validate.py  # must report PASS
python spec/sim.py       # must report PASS

validate.py is fully offline: the stable external W3C contexts (credentials/v2, data-integrity/v2) are vendored under spec/contexts/ and served by a local document loader, so JSON-LD expansion does not depend on network access to w3.org.

Harness modules

spec/avp_crypto.py
P-256 key derivation, JCS canonicalization ([[RFC8785]]), ecdsa-jcs-2022 sign/verify (deterministic RFC 6979, canonical low-s). The mandatory-to-implement cryptosuite.
spec/sdjwt.py
P-256 key derivation, ES256/JOSE sign/verify, JWK encode/decode, SD-JWT compact serialization, make_disclosure, sd_hash, and key-binding JWT helpers. Used only by the interop bundle.
spec/interop.py
AVP-Micro ↔ SD-JWT-VC translator: claim mapping, three bridge modes, cross-stack verification rules, importAdvisory surfacing, and L3 key-binding JWT.
spec/pricing.py
Normative pricing-model evaluator: flat, per-call, tiered, and composite models. Used by the Payments bundle conformance vectors.
spec/sim.py + spec/sim-scenarios.json
Reference protocol simulator and its declarative use cases.

Protocol simulator & interactive demo

spec/sim.py runs the full signed-message flow end to end — quote → authorize → execute → receipt, plus streaming/metered sessions, human-present approval, AP2-bridged imported authority, and the refund/reversal/dispute lifecycle — with real ecdsa-jcs-2022 proofs and complete wallet-side policy enforcement, settled on a simulated play-money ledger. Settlement is the only money-touching step and is the one part the protocol scopes out, so it is stubbed: no real funds move.

The use cases are defined declaratively in spec/sim-scenarios.json and cover the happy paths plus the full rejection vocabulary (caps, replay, expiry, quote-binding, budget, forged approval, over-refund, …). An interactive Streamlit demo (separate avp-micro-sim-demo repository) visualises each use case: a plain-language outcome, a flow diagram of who signs what, the money moving on the ledger, and a per-participant view. 

python spec/sim.py    # PASS/FAIL per declarative use case

Relationship to network agent-payment schemes

AVP-Micro is the trust and authorization layer, not a payment network. It standardizes who may spend, under what terms, and how that authority is proven, leaving value movement to any settlement rail.

Emerging network schemes for agent payments use a different credential stack: Mastercard Agent Pay for Machines / Verifiable Intent and Google AP2 are built on [[SD-JWT-VC]] — three-layer KB-SD-JWT chains signed with ES256, typed by vct, and identified by iss/sub URIs. They deliberately do not conform to the W3C VC Data Model and do not use DIDs. AVP-Micro, by contrast, secures W3C VC 2.0 credentials with Data Integrity proofs and identifies parties by DID.

The SD-JWT-VC Interop Profile bridges this gap normatively. See the bridge design note for the full analysis of the three bridge modes and identity asymmetry.
