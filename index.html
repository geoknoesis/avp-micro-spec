SD-JWT-VC Interop Profile

avp-micro/interop/sd-jwt-vc/v1#

A normative bridge between AVP-Micro and the SD-JWT-VC credentials used by Mastercard/Google Verifiable Intent and Google AP2. Three bridge modes; no silent trust escalation.

  • Proof-preserving (default) — embed + verify both
  • Co-issued — native DI proof + parallel ES256
  • Attested — named bridge re-signs
  • L3 key-binding JWT (per-purchase action)
  • importAdvisory for lossless surfacing
  • BitstringStatusList ↔ Token Status List

Positioning

What this layer adds

AVP-Micro is the trust and authorization layer, not a payment network. It standardizes who may spend, under what terms, and how that authority is proven — and leaves value movement to any settlement rail.

🏛️

Vendor-neutral trust anchor

Authority is proven by verifying an issuer's credential against a DID — not by trusting one network's ledger, token format, or settlement guarantee.

🔌

Portable across rails

One credential grammar spans competing schemes. An agent's spending authority is not locked to a single network's substrate — Lightning, Interledger, card, or stablecoin.

📋

Standards-based, not proprietary

Built entirely on ratified W3C specs (eddsa-jcs-2022, BitstringStatusList) with signed test vectors — no dependency on a network operator's SDK.

🔁

Bridge to SD-JWT-VC networks

The interop profile maps credentials to and from Mastercard Agent Pay for Machines / Google AP2 without collapsing the two trust stacks.

Interoperability

Relationship to network schemes

Emerging network schemes for agent payments each have a built-in credential stack. AVP-Micro treats them as settlement substrates it can sit above, bridging at the authorization layer.

Property AVP-Micro Mastercard Agent Pay / Verifiable Intent Google AP2
Credential format W3C VC 2.0 / JSON-LD SD-JWT VC (IETF) SD-JWT VC (IETF)
Proof mechanism Data Integrity eddsa-jcs-2022 JOSE ES256 (KB-SD-JWT) JOSE ES256

W3C VCDM 2.0 ✔ normative informative reference only informative reference only
Settlement rail any (pluggable) card / stablecoin (Polygon, Solana, Base) card / x402
Interop with AVP-Micro — ⇄ bridge required (see interop profile) ⇄ bridge required (see interop profile)

A W3C Data Integrity proof signs JCS-canonical JSON-LD bytes; an SD-JWT signature signs JOSE serialization. Translating claims changes the signed bytes — the original signature can never verify over the translated form. The interop profile bridges this gap without collapsing either trust stack. Conformance harness

Run the harness

A shared Python harness at spec/ root generates all signed test vectors and validates every bundle. All checks are offline and deterministic.

1

Install dependencies

python -m venv .venv
.venv/Scripts/activate   # Windows
# or: source .venv/bin/activate
pip install -r requirements.txt
2

Generate signed test vectors

Deterministic key derivation — running this twice produces byte-identical output.

python spec/generate.py
3

Verify proofs and bindings

Crypto round-trip, policy checks, interop A↔V round-trip, tamper-detection negatives.

python spec/verify.py    # must report PASS
4

Validate artifacts

Turtle parse, offline JSON-LD expansion, JSON Schema, and SHACL for all three bundles.

python spec/validate.py  # must report PASS

Harness modules

avp_crypto.py

Ed25519 key derivation, JCS canonicalization, eddsa-jcs-2022 sign/verify. Mandatory-to-implement cryptosuite.

sdjwt.py

P-256 keys, ES256/JOSE, JWK, SD-JWT compact primitives and key-binding JWT helpers for the interop bundle.

interop.py

AVP-Micro ↔ SD-JWT-VC translator: claim mapping, three bridge modes, cross-stack verification, importAdvisory.

pricing.py

Pricing-model evaluator: flat, per-call, tiered, composite — used by the Payments conformance vectors.

Per-bundle artifacts

What each bundle contains

Every bundle is self-contained and follows the same artifact layout.

📄 W3C ReSpec document

Normative HTML specification rendered client-side by ReSpec.js. Status: unofficial.

🗂️ JSON-LD 1.1 context

Normative context mapping terms to IRIs. @protected: true, version 1.1.

✅ JSON Schema

Draft 2020-12 schema with $defs per type. Conformance aid; validates every test vector.

🔷 SHACL shapes

RDF shape constraints. Each vector passes SHACL validation in validate.py.

🦉 RDFS / OWL ontology

Turtle vocabulary declaring classes and properties in the bundle's namespace.

🧪 Signed test vectors

Deterministic JSON fixtures with live cryptographic proofs. Regenerated by generate.py.
